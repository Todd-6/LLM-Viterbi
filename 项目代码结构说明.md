本文档用于说明这个仓库准备上传到 GitHub 时，代码各部分分别在做什么、彼此之间的关系是什么，以及代码里实际使用到的“调用地址/路径”有哪些。

项目根目录：

仓库根目录（repository root）

## 1. 这个项目整体是在做什么

这个项目的核心目标，是把英文文本转成 bit 序列，通过卷积码编码和 BPSK 调制后加入噪声，再尝试用不同的译码方式把文本恢复出来，并比较它们在不同 SNR 条件下的表现。

仓库里主要比较了 3 类方案：

1. `标准 Viterbi 译码`
2. `标准 Viterbi + 语言模型纠错`
3. `带语言模型剪枝的 Viterbi 译码`

其中真正的核心算法几乎都集中在 `decoder/viterbi_lm_decode.py` 里，其他大多数文件本质上都是“批量测试脚本”或“结果可视化脚本”。

## 2. 目录和文件分别做什么

### `decoder/`

核心算法目录。

- `decoder/viterbi_lm_decode.py`
  - 这是整个项目最重要的文件。
  - 负责：
    - 文本转 bit：`text_to_binary()`
    - bit 转文本：`binary_to_text()`
    - 卷积编码：`convolutional_encode()`
    - BPSK 调制：`modulate_bpsk()`
    - 加噪：`add_noise()`
    - 标准维特比译码：`viterbi_decode_standard()`
    - 带语言模型参与剪枝的维特比译码：`viterbi_decode_with_lm()`
    - 标准译码后的语言模型纠错：`correct_text_with_lm()`
    - 语言模型初始化：`initialize_lm()`
    - 纠错模型初始化：`initialize_correction_lm()`
  - 这个文件既可以被别的脚本导入，也可以单独运行，用于演示完整流程。

### `evaluation/`

评测脚本目录，主要负责批量测试、统计和性能比较。

- `evaluation/batch_test_bler.py`
  - 主批量评测脚本之一。
  - 从测试句库里随机抽句子，在多个 SNR 下比较：
    - 标准 Viterbi
    - 标准 Viterbi + Correction LM
    - LM-Viterbi
  - 输出文本结果到 `results/batch_test_results_时间戳.txt`

- `evaluation/batch_collect_standard.py`
  - 只跑标准 Viterbi。
  - 输出每一条样本的原句和译码结果，方便人工逐行查看。
  - 输出到 `results/batch_standard_viterbi_results.txt`

- `evaluation/batch_test_sbert.py`
  - 在常规测试基础上，再引入 SBERT 语义相似度评分。
  - 会记录：
    - 标准 Viterbi 结果
    - 标准 Viterbi + 纠错结果
    - LM-Viterbi 结果
    - 各自 SBERT 分数
    - 各自耗时
  - 输出到 `results/batch_test_results_sbert.csv`

- `evaluation/batch_test_interval_sweep.py`
  - 专门扫描 `LM_CONTEXT_CHECK_INTERVAL` 这个参数。
  - 也就是测试：每隔多少字符调用一次 LM 做路径剪枝更合适。
  - 输出到：
    - `results/interval_sweep_results_时间戳.txt`
    - `results/interval_sweep_progress_时间戳.txt`

- `evaluation/batch_test_performance.py`
  - 做性能/耗时对比。
  - 比较：
    - 标准 Viterbi + 纠错
    - LM-Viterbi
  - 默认主要把结果打印到终端，不是主要的文件输出脚本。

- `evaluation/batch_time_test_performance.py`
  - 作用与上面类似，也是性能测试脚本。
  - 结构几乎相同，重点在耗时 benchmark。

### `scripts/`

结果处理和可视化目录。

- `scripts/plot_bler_results.py`
  - 读取 `results/` 目录下最新的 `batch_test_results_*.txt`
  - 解析每个 SNR 的错误率
  - 画出 BLER 曲线图
  - 输出到 `results/bler_plot.png`

### `data/`

输入数据目录。

- `data/clean_sentences_test_set.txt`
  - 当前评测脚本最主要的测试句库。
  - `evaluation` 目录下多个脚本都会从这里读取测试文本。

- `data/byt5_training_data_full.jsonl`
- `data/byt5_correction_train_data.jsonl`
  - 从当前已检查的源码来看，这两个文件更像训练数据或历史数据。
  - 在目前这些评测脚本里没有直接被调用。

### `models/`

本地模型目录。

- `models/byt5_finetuned`
  - 提供给 LM-Viterbi 剪枝阶段使用的 ByT5 模型。

- `models/ByT5_correction_finetuned`
  - 提供给标准 Viterbi 后处理纠错使用的 ByT5 模型。

### `results/`

结果输出目录。

- 存放批量测试结果、CSV、绘图结果。

### `debug/`

调试输出目录。

- `decoder/viterbi_lm_decode.py` 会在这里写入一些调试文件，例如：
  - `lm_debug_output.txt`
  - `step_debug_log_forSingle.txt`
  - `standard_step_debug_log.txt`
  - `lm_detailed_calls.csv`

### `requirements.txt`

项目依赖文件，当前列出的核心依赖包括：

- `torch`
- `transformers`
- `numpy`
- `scikit-learn`
- `sentence-transformers`
- `tqdm`
- `matplotlib`
- `pandas`

## 3. 最核心的调用关系

可以把这个项目理解成下面这条主线：

`evaluation/*.py` 或 `decoder/viterbi_lm_decode.py`  
-> 调用 `decoder/viterbi_lm_decode.py` 里的核心函数  
-> 读取 `data/clean_sentences_test_set.txt` 或内置测试文本  
-> 使用 `models/` 里的本地模型  
-> 把测试结果写到 `results/` 或 `debug/`

更具体一点：

### 主核心模块

- `decoder/viterbi_lm_decode.py`
  - 是整个项目的算法中心。
  - 几乎所有评测脚本都要依赖它。

### 评测脚本与核心模块关系

- `evaluation/batch_test_bler.py`
  - `import viterbi_lm_decode as vlm`
  - 调用：
    - `vlm.initialize_lm()`
    - `vlm.initialize_correction_lm()`
    - `vlm.text_to_binary()`
    - `vlm.convolutional_encode()`
    - `vlm.modulate_bpsk()`
    - `vlm.add_noise()`
    - `vlm.viterbi_decode_standard()`
    - `vlm.correct_text_with_lm()`
    - `vlm.viterbi_decode_with_lm()`

- `evaluation/batch_collect_standard.py`
  - `import viterbi_lm_decode as viterbi`
  - 只走标准译码链路：
    - `text_to_binary()`
    - `convolutional_encode()`
    - `modulate_bpsk()`
    - `add_noise()`
    - `viterbi_decode_standard()`

- `evaluation/batch_test_sbert.py`
  - 直接从 `viterbi_lm_decode` 导入多个函数
  - 同时额外调用：
    - `SentenceTransformer('all-MiniLM-L6-v2')`
  - 相当于：
    - 跑三种译码结果
    - 再用 SBERT 算语义相似度

- `evaluation/batch_test_interval_sweep.py`
  - `import viterbi_lm_decode as vlm`
  - 主要调用：
    - `initialize_lm()`
    - `viterbi_decode_with_lm()`
  - 专门测试 LM 剪枝调用频率的影响

- `evaluation/batch_test_performance.py`
- `evaluation/batch_time_test_performance.py`
  - 都是：
    - 先准备信号
    - 再比较 `标准 Viterbi + correct_text_with_lm()` 和 `viterbi_decode_with_lm()` 的耗时

- `scripts/plot_bler_results.py`
  - 不参与译码
  - 只负责读取已有结果文件并画图

## 4. 最重要的“调用地址/路径”是什么

这里的“地址”我分成 4 类来说明：

1. 模块导入地址
2. 数据输入地址
3. 模型加载地址
4. 结果输出地址

### 4.1 模块导入地址

#### 明确插入 `decoder` 路径的脚本

这些脚本运行时会主动把 `decoder/` 加入 Python 导入路径，所以更稳定：

- `evaluation/batch_test_bler.py`
  - 插入：`../decoder`

- `evaluation/batch_test_performance.py`
  - 插入：`../decoder`

- `evaluation/batch_time_test_performance.py`
  - 插入：`../decoder`

#### 导入路径写法不太统一的脚本

这些脚本里没有统一地显式加入 `../decoder`，所以运行时更依赖当前工作目录或 IDE 配置：

- `evaluation/batch_collect_standard.py`
  - 只做了：`sys.path.append(current_dir)`
  - 但随后写的是：`import viterbi_lm_decode as viterbi`
  - 这意味着它默认假设 `viterbi_lm_decode.py` 已经在 Python 可见路径中

- `evaluation/batch_test_interval_sweep.py`
  - 同样只做了：`sys.path.append(current_dir)`
  - 然后直接 `import viterbi_lm_decode as vlm`

- `evaluation/batch_test_sbert.py`
  - 直接 `from viterbi_lm_decode import ...`
  - 没有显式插入 `../decoder`

这 3 个脚本在 GitHub 给别人使用时，建议后续统一成和 `batch_test_bler.py` 一样的导入方式，否则可能出现导入失败。

### 4.2 数据输入地址

当前最主要的数据输入文件是：

- `data/clean_sentences_test_set.txt`

在各脚本中的实际写法包括：

- `evaluation/batch_test_bler.py`
  - 使用绝对化后的：`../data/clean_sentences_test_set.txt`

- `evaluation/batch_collect_standard.py`
  - 使用绝对化后的：`../data/clean_sentences_test_set.txt`

- `evaluation/batch_test_sbert.py`
  - 使用绝对化后的：`../data/clean_sentences_test_set.txt`

- `evaluation/batch_test_interval_sweep.py`
  - 直接写成：`../data/clean_sentences_test_set.txt`

- `evaluation/batch_test_performance.py`
  - `SENTENCE_FILE = "../data/clean_sentences_test_set.txt"`

- `evaluation/batch_time_test_performance.py`
  - `SENTENCE_FILE = "../data/clean_sentences_test_set.txt"`

### 4.3 模型加载地址

这些地址定义在 `decoder/viterbi_lm_decode.py` 里。

#### LM-Viterbi 剪枝模型

- `LM_MODEL_NAME_OR_PATH = "./models/byt5_finetuned"`
- 回退模型：`LM_MODEL_FALLBACK = "google/byt5-small"`

含义：

- 优先从本地加载 `models/byt5_finetuned`
- 如果本地模型加载失败，就尝试从 HuggingFace 模型名 `google/byt5-small` 加载

#### 标准译码后处理纠错模型

- `CORRECTION_MODEL_PATH = "./models/ByT5_correction_finetuned"`

#### SBERT 模型

- `SentenceTransformer('all-MiniLM-L6-v2')`

这个 SBERT 模型出现在：

- `evaluation/batch_test_sbert.py`
- `decoder/viterbi_lm_decode.py` 的语义评估相关部分

### 4.4 结果输出地址

#### 批量测试结果

- `evaluation/batch_test_bler.py`
  - 输出：`results/batch_test_results_时间戳.txt`

- `evaluation/batch_collect_standard.py`
  - 输出：`results/batch_standard_viterbi_results.txt`

- `evaluation/batch_test_sbert.py`
  - 输出：`results/batch_test_results_sbert.csv`

- `evaluation/batch_test_interval_sweep.py`
  - 输出：`results/interval_sweep_results_时间戳.txt`
  - 进度文件：`results/interval_sweep_progress_时间戳.txt`

#### 绘图结果

- `scripts/plot_bler_results.py`
  - 输出：`results/bler_plot.png`

#### 调试输出

由 `decoder/viterbi_lm_decode.py` 写入：

- `debug/lm_debug_output.txt`
- `debug/step_debug_log_forSingle.txt`
- `debug/standard_step_debug_log.txt`

## 5. 核心算法链路怎么走

仓库中最标准的一条单样本处理链路如下：

1. 输入一句英文文本
2. 调用 `text_to_binary()` 转成 bit 序列
3. 调用 `convolutional_encode()` 做卷积编码
4. 调用 `modulate_bpsk()` 做 BPSK 调制
5. 调用 `add_noise()` 按设定 SNR 加噪
6. 进入两条或三条比较分支：
   - `viterbi_decode_standard()`
   - `viterbi_decode_standard()` 后接 `correct_text_with_lm()`
   - `viterbi_decode_with_lm()`
7. 把结果和原句比较，统计错误率、耗时、语义相似度等
8. 输出到 `results/`

## 6. 这个仓库里有没有远程 API 地址

从当前检查到的核心源码来看：

- 没有发现 `OpenAI API`、`base_url`、HTTP 接口 URL 之类的显式远程服务地址
- 主要依赖的是：
  - 本地模型目录 `./models/...`
  - HuggingFace `from_pretrained(...)`

也就是说，这个项目目前不是“调用某个在线接口”的结构，而是“本地代码 + 本地/Hub 模型加载”的结构。

如果本地模型不存在，下面这些模型名可能会触发 HuggingFace 下载或缓存读取：

- `google/byt5-small`
- `all-MiniLM-L6-v2`

## 8. 当前代码关系的简化结论

如果只用一句话概括：

这个仓库是一个“以 `decoder/viterbi_lm_decode.py` 为核心，多个 `evaluation/*.py` 脚本围绕它做误码率、语义相似度和耗时测试，再由 `scripts/plot_bler_results.py` 对结果进行可视化”的项目。

如果只看最关键的依赖关系：

- 算法核心：`decoder/viterbi_lm_decode.py`
- 测试输入：`data/clean_sentences_test_set.txt`
- 本地模型：`models/byt5_finetuned`、`models/ByT5_correction_finetuned`
- 测试输出：`results/`
- 调试输出：`debug/`

## 9. 我帮你额外识别到的一个注意点

当前 `evaluation/` 里不同脚本对 `viterbi_lm_decode.py` 的导入方式不完全一致。

这意味着：

- 你自己在本地 IDE 里可能能跑
- 但别人从 GitHub 拉下来后，直接运行某些脚本时，可能会遇到 `ModuleNotFoundError: No module named 'viterbi_lm_decode'`

1. 帮你再输出一份更适合放 GitHub 的 `README.md` 项目说明
2. 帮你把 `evaluation/` 里所有脚本的导入路径统一整理好，避免别人运行时报错
